# reDonate Bot — redonate.su

Telegram-магазин игрового доната. Стек: **aiogram 3** + **FastAPI** + **React/Vite** + **PostgreSQL** + **Redis/Celery**.

## Workflow

Локально — только разработка кода (черновик). Docker, запуск бота, БД — только на сервере.

```
пишем код → git push → GitHub Actions → автодеплой на VPS (185.23.19.227)
```

Никогда не предлагать `make dev`, `docker compose up` и т.п. — локально это не используется.

## Структура

```
bot/           — Telegram Bot (aiogram 3)
  handlers/    — client/ и admin/ хендлеры
  utils/texts.py — все тексты бота (используют SHOP_NAME из .env)
api/           — FastAPI backend
  routers/, services/, schemas/
miniapp/       — React + Vite (Telegram Mini App)
worker/        — Celery задачи
shared/
  config/settings.py — все настройки (Pydantic Settings)
  models/            — SQLAlchemy модели
migrations/    — Alembic
```

## Стек и стиль

- **Python**: asyncio, Pydantic v2, SQLAlchemy 2.0 async, aiogram 3.x
- **FastAPI**: lifespan, async везде, dependency injection
- **React**: функциональные компоненты + хуки, Vite, Tailwind + shadcn/ui или Radix UI
- **UI**: современный минималистичный дизайн, Framer Motion для анимаций
- Не использовать устаревшие паттерны (классовые компоненты, sync ORM, старый роутинг)

## Правила кода

- Все тексты бота — только в `bot/utils/texts.py`, SHOP_NAME только из `.env`
- Миграции — только через Alembic, никогда не менять схему вручную
- Линтер/форматтер: `ruff` (установлен локально, хук запускает автоматически)
- Секреты — только через `shared/config/settings.py`, никогда не хардкодить
- Бизнес-логика — только в `api/services/`, не в роутерах и не в bot-хендлерах
- Bot-хендлеры содержат ноль бизнес-логики — только вызов API и форматирование ответа

## Агенты

В проекте есть специализированные агенты в `.claude/agents/`. Claude Code вызывает их автоматически через инструмент Task.

| Агент    | Когда вызывается                                              |
|----------|---------------------------------------------------------------|
| teamlead | Любая задача на 2+ слоя, или если непонятно к кому идти      |
| backend  | api/, shared/models/, migrations/, бизнес-логика             |
| bot      | bot/, хендлеры, FSM, клавиатуры, тексты                      |
| ui-ux    | miniapp/, компоненты, экраны, анимации                        |
| devops   | .github/workflows/, Dockerfile, docker-compose                |
| reviewer | После любых изменений кода, перед финальным ответом           |
| security | Задачи с оплатой, токенами, пользовательским вводом           |

**Порядок для типичной фичи:** teamlead → backend → bot/ui-ux → reviewer → security (если платёжное)

## Диагностика (читай перед любым фиксом)

### Главный принцип
Фикс делается на уровне владельца проблемы (owner layer), а не там где появился симптом.

### Перед любым изменением кода определи:
```
Root cause:      [что именно сломано]
Owner layer:     [bot / api / miniapp / shared / migrations]
Затронуто слоёв: [перечисли все]
Риск:            Low / Medium / High
```

### Трейс для поиска root cause

Top-down (от входа к данным):
```
Telegram update → bot handler → API call → FastAPI router → service → repository → SQLAlchemy → PostgreSQL
```

Bottom-up (от данных к выходу):
```
PostgreSQL → model → service → API response → bot handler → reply / Mini App UI
```

Диагностируй в порядке: `данные/контракты → бизнес-логика → async/timing → UI state → интеграция`

### Правила диагностики

- Симптом в боте → сначала проверь API контракт и сервис
- Симптом в UI → сначала проверь hook и API response
- Симптом в роутере → сначала проверь сервис
- Не патчи симптом в дочернем слое если root cause в родительском

### Масштаб изменений

| Файлов затронуто | Действие                                               |
|------------------|--------------------------------------------------------|
| 1 файл           | Обоснуй письменно почему остальные не затронуты        |
| 2–3 файла        | Норма — проверь согласованность контрактов             |
| 4+ файлов        | Сообщи план пользователю до начала                     |
| Реархитектура    | **Стоп. Только с явного одобрения пользователя.**      |

### При изменении контракта — обновить все связанные слои:
- [ ] Pydantic schema (api/schemas/)
- [ ] SQLAlchemy модель (shared/models/) + Alembic миграция
- [ ] Сервисы, которые используют модель
- [ ] Bot-хендлеры, которые вызывают этот эндпоинт
- [ ] UI-компоненты и хуки, которые потребляют этот API

## Требует одобрения пользователя — полная остановка

- Реархитектура любого слоя
- Деструктивные миграции Alembic (drop column, rename table)
- Изменение логики аутентификации/авторизации
- Изменения CI/CD pipeline или Docker-конфигов
- Любое изменение затрагивает 4+ файлов в разных слоях

## Антипаттерны — никогда не делать

- ❌ Патчить симптом в дочернем слое не разобравшись с родительским
- ❌ Бизнес-логика в bot-хендлерах или FastAPI роутерах
- ❌ Тексты бота вне `bot/utils/texts.py`
- ❌ Секреты захардкожены в коде
- ❌ Прямой доступ бота к БД (только через API)
- ❌ Изменение схемы БД без Alembic миграции
- ❌ `except Exception: pass` — тихое проглатывание ошибок
- ❌ Предлагать `docker compose up` локально
